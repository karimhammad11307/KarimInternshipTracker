📦 PuckConfigTest: Architecture & Environment Deep Dive

> A sandbox project to deconstruct the Next.js and Puck visual builder architecture, focusing on the "Config vs. Representation" pattern.

## 🎯 Objective
To implement and analyze a lightweight e-commerce prototype using **Next.js 15+** and the **Puck** visual page builder. The primary goal is to practically explore how visual builders operate entirely independently of backend logic by utilizing a strict **JSON-in, JSON-out** flow.

---

## 🏗️ Project Architecture & Routing

Our Next.js App Router topology is structured to strictly separate server-side data fetching from client-side interactive rendering:

* **`puck.config.tsx` (The Blueprint):** The single source of truth. It maps string types (e.g., `"ProductCard"`) to React schemas and explicitly defines the input fields (text, select) required to populate the component properties.
* **`src/app/[...puckPath]/page.tsx` (Server Component):** Utilizes Next.js "Catch-All Routing" (`[...]`) to intercept incoming URLs. It securely awaits asynchronous `params`, determines whether the route is the editor (`/edit`) or the live site, and passes the initial JSON state downwards.
* **`src/app/[...puckPath]/client.tsx` (Client Component):** Marked with `"use client"`, this file bridges the server and browser. It mounts the interactive `<Puck>` drag-and-drop engine and handles the `onPublish` event to serialize the canvas into a stringified JSON payload.
* **`src/app/globals.css`:** Stripped of default styles (like Next.js dark mode overrides) to ensure the Puck editor's built-in CSS UI library functions without visual interference.

## 🧠 Core Front-End Concepts Validated

### 1. Separation of Concerns (Config vs. Representation)
We proved that the database never needs to store HTML or CSS. 
* **The Representation** is purely an array of JSON objects (e.g., `{"type": "ProductCard", "props": {"productName": "T-Shirt", "price": 25}}`).
* **The Configuration** stays safely inside the Git codebase as React components.

### 2. State Management & Dynamic UI Updates
Edits made in the sidebar instantly mutate Puck's internal JSON state. React detects this state mutation and triggers a **re-render** of the central canvas, instantly painting the new props into the component's HTML wrapper without requiring a page refresh.

### 3. The HTTP Request & Render Cycle
Traced the request lifecycle: The server handles the initial route request and passes an empty representation state (`initialData`). The browser then takes over, executing the bundled JavaScript to mount the interactive editor sidebars and canvas.

### 4. Asynchronous Data Handling (Next.js 15)
Resolved Next.js 15 Promise-based routing mechanics. URL `params` are awaited in the Server Component before unwrapping route segments, preventing race conditions during dynamic route generation.
