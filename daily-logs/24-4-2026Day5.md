Objective
Evaluate potential visual editor libraries (specifically Puck vs. Craft.js) to satisfy the requirement of "splitting config from representation" for our B2B SaaS Shopify-like platform.

Core Architectural Concept: "Config vs. Representation"
To replicate Shopify's theme architecture, the editor must act strictly as a visual layer over a JSON data model.

Representation: A pure JSON tree defining sections, blocks, and props (e.g., sections/*.json).

Config: The React components and field schemas that read the JSON.

The editor internal state must never be trapped in React; it must be JSON in, JSON out, allowing our Rust backend to store, validate, and version the data completely independently.

Tool Evaluation
1. Puck (Recommended)
Architecture: Ships with a true JSON-first, "config vs. representation" pattern out-of-the-box.

Data Model: Its JSON schema naturally mirrors Shopify's sections and blocks structure.

Team Fit: Highly efficient for a 2-person Front-End team. We can focus on building UX and components rather than engineering a drag-and-drop tree engine from scratch.

Extensibility: We can use Puck as the core merchant-facing editor and easily fork it later to add custom schema validations or multi-store features.

2. Craft.js (Overkill for V1)
Architecture: A state-tree library for nodes and props. It is incredibly powerful but does not come with a predefined JSON schema or UI.

Drawback: We would have to build the entire editor UI, the serialization logic, and the Shopify-style section wrapper from scratch.

Verdict: Unnecessary for our current phase. We can revisit Craft.js later for advanced internal developer tooling if we outgrow Puck's internal drag-and-drop mechanics.

Infrastructure Impact (JSON Size & Storage)
Footprint: The JSON files generated by these editors are negligible in size (typically 1–10 KB per page/theme).

Storage Strategy: Our Rust backend will easily handle storing one JSON blob per theme, per store. This lightweight footprint makes versioning (e.g., theme_v1.json, theme_v2.json) and caching trivial. The actual performance bottleneck will be the frontend React rendering, not the JSON storage.
