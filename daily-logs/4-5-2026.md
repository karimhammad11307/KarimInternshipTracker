# Architecture V3 Review (4-5-2026)

## Background
My teammate and I debated various architectural approaches extensively before settling on this V3 architecture. We needed to balance rendering accuracy with a highly responsive user experience.

## The Dual-Model Architecture
This system separates the data from the visual canvas:
1. **Metadata JSON (Source of Truth)**: Stored in React (Zustand), saved to the database.
2. **GrapesJS Canvas (Visual Model)**: An iframe that displays the HTML and handles interactions.

## Core Concepts
*   **data-eid**: HTML attribute linking a visual element to its JSON data block.
*   **MetadataStore**: Zustand store holding the raw JSON data.
*   **CommandManager**: Handles Undo/Redo operations and builds an action history.
*   **SyncEngine**: Bridges the MetadataStore and the canvas (Push/Pull data).
*   **pushDepth**: A counter in SyncEngine using `queueMicrotask` to prevent infinite event loops.

## Event Data Flow
Data syncs only at "completion points" (e.g., when typing finishes).

*   **Push (Sidebar Edit)**: React UI -> `CommandManager` -> `MetadataStore` -> `SyncEngine` -> Canvas updates instantly via CSS injection.
*   **Pull (Canvas Edit)**: Canvas interaction ends -> `SyncEngine` sets `pushDepth > 0` -> updates `MetadataStore` -> blocks the echo push -> `pushDepth` resets.

## Rendering Approaches Comparison

### 1. Local DOM Manipulation (Architecture V3 - Selected)
The `SyncEngine` mimics changes locally by directly injecting CSS or text into the DOM.
*   **Pros**: Zero latency (< 5ms). Smooth real-time experience (allows dragging sliders).
*   **Cons**: High frontend logic complexity.

### 2. Server Round-Trip (Deprecated)
A user makes an edit, JSON is sent to the server, and the full HTML is rendered and returned.
*   **Pros**: 100% accurate to backend templates. Simpler frontend code.
*   **Cons**: Unusable lag (200ms+). Screen flickering. Loss of focus and scroll state. 
*   *Note: We only use this now for "Structural Changes" (adding/removing blocks).*
